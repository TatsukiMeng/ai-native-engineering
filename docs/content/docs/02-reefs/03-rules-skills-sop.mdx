# 这一章要解决什么问题

AI 协作里最浪费的事情之一，就是同一个坑反复踩、同一个 Prompt 反复重写、同一种任务每次都重新解释。

如果经验不能沉淀，所谓“提效”往往只是一时的。今天跑通一次，明天换个任务、换个模型、换个上下文，结果又从头开始。

# 这一章会怎么展开

- 先区分什么应该写进 Rules，什么应该沉淀成 Skills，什么应该升级成 SOP
- 再讨论它们分别适合约束什么层面：风格、流程、命令、验证还是协作边界
- 最后说明这些资产怎样反过来降低波动、减少返工并提升团队协作一致性

:::note[占位说明]
这篇后续会补入 `AGENTS.md`、项目记忆、Skill 模板、任务 SOP 与回写机制的具体案例。
:::

# 读完这一章，读者应该带走什么

- 偶发成功不等于方法论，只有沉淀下来才算资产
- Rules、Skills、SOP 不是一回事，适用层级也不同
- 真正稳定的交付，背后一定有一套可复用的控制面配置
