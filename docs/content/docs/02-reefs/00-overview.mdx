# 这一部分为什么要先讲“怎么死”

很多团队一谈 AI 提效，就急着开干：先让它写页面，再让它补接口，再让它自己跑测试，最后希望它能一路丝滑把项目推到上线。

问题是，真实世界不是 demo 场。

项目一旦变大，最先冒出来的往往不是“不会生成”，而是**调试越来越痛、验证越来越假、经验无法复用、失败也收敛不住**。

所以 Part 2 不负责唱高调，它负责提前把这些暗礁摊开讲。

# 本部分的阅读地图

- `第 1 章` 讲为什么开发更快了，调试反而更痛苦
- `第 2 章` 讲为什么最危险的不是报错，而是 AI 开始自洽地骗过自己
- `第 3 章` 讲 Rules、Skills 和 SOP 怎么把偶发成功变成稳定交付
- `第 4 章` 讲熔断、回滚、审计与小步快跑这套工程护栏怎么搭

:::warning[别把“能跑起来”误判成“工程上安全”]
AI 时代最大的坑之一，就是把一次看起来顺利的生成，当成了长期可复用的方法。

这一部分的任务，就是先把这种错觉打掉。
:::

# 读完这一部分，至少应该建立什么底线

- 不能让 AI 既当开发又当唯一裁判
- 不能把关键经验只留在聊天记录里
- 不能没有熔断、回滚和审计就直接大步推进
