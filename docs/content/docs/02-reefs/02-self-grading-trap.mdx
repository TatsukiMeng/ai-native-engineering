# 这一章要解决什么问题

很多人第一次碰到 AI 生成代码时，最警惕的是报错。

但真实项目里，更危险的往往不是报错，而是它**看起来解释得很顺、测试也像是自己补齐了、日志也像说得通，最后却把错误结论包成了一套自洽叙事**。

这章要拆的，就是这种“假稳定”。

# 这一章会怎么展开

- 先拆 AI 为什么会在错误前提下继续生成一整套自圆其说的结果
- 再拆为什么让 AI 既写实现、又补测试、还自己解释问题，会特别危险
- 最后说明工程上该怎么设计交叉验证、人工抽检和异源材料校验

:::warning[占位说明]
这篇先完成章节占位，后续会补入真实翻车案例、自测自洽链路、以及如何设计不让 AI 自评闭环吞掉风险。
:::

# 读完这一章，读者应该带走什么

- 不能把“解释得通”当成“已经做对”
- AI 不适合做唯一裁判，尤其不适合给自己盖章验收
- 验证链路必须引入独立视角，而不是同源材料反复循环
