# 这一章要解决什么问题

项目一开始最容易犯的错，不是技术选型错，而是**需求还没收敛，就急着让 AI 开写**。

这样做短期会显得很快，长期几乎一定会返工：边界变来变去，接口反复重做，页面和后端互相打架，最后 AI 写得越多，清理成本越高。

# 这一章会怎么展开

- 先讲需求为什么要先收敛成可执行的项目边界
- 再讲哪些信息应该沉淀成 Spec、任务列表和验收标准
- 最后讲项目启动阶段，人类该盯什么，AI 又该从哪里介入最合适

:::tip[占位说明]
这篇后续会补入需求收敛模板、Kickoff Prompt、任务拆解样例和“什么暂时不要做”的裁剪策略。
:::

# 读完这一章，读者应该带走什么

- 项目启动阶段最重要的不是写得快，而是边界清楚
- 没有收敛需求就直接生成，返工几乎是注定的
- AI 最适合在边界明确后加速，而不是替代项目定义本身
