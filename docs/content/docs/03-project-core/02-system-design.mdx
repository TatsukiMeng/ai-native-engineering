# 这一章要解决什么问题

真实项目一旦进入实现阶段，最怕的不是某个函数写得慢，而是没有结构骨架：模块怎么拆、数据怎么流、接口怎么定、边界怎么守，全都边写边猜。

AI 在这种状态下也很容易“局部看起来合理，整体却越来越乱”。

# 这一章会怎么展开

- 先讲架构图、数据模型、接口契约和模块边界为什么必须前置
- 再讲哪些设计材料适合长期放进 Context，哪些应该转成文档与规则
- 最后讲系统设计阶段怎么让 AI 参与，但不把架构判断完全外包出去

:::note[占位说明]
这篇后续会补入系统设计清单、数据与接口设计样例、模块边界检查表，以及设计评审时的人机分工方式。
:::

# 读完这一章，读者应该带走什么

- 系统设计不是文档装饰，而是后续 AI 协作的路由表
- 边界越早立住，后面的生成越稳定
- 让 AI 参与设计可以，但最终结构判断必须可审查、可复盘
