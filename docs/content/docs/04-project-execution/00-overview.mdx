# 这一部分补的不是功能，而是执行力

当项目主干已经立住，下一步最容易犯的错，就是继续把注意力全放在“多生成功能”。

但真实交付里，真正决定效率上限和稳定性的，往往不是再多写几个页面，而是：**终端执行能力、外部系统接入、浏览器自动化、跨端宿主约束，以及最终的联调发布闭环。**

这也是 Part 4 要做的事：给项目装上更强的手和眼。

# 本部分的阅读地图

- `第 1 章` 讲 CLI 与 Harness，为什么高价值任务最终都要落到终端和受控执行环境
- `第 2 章` 讲 MCP 与外部系统接入，怎么少让 AI 瞎猜，多让它直接读真实材料
- `第 3 章` 讲 Playwright 与多模态执行，怎么让 AI 真正有眼睛和手
- `第 4 章` 讲跨端与宿主，项目不只跑在浏览器里时控制面要怎么升级
- `第 5 章` 讲联调、验收与发布，为什么代码写完远远不等于工作流闭环

:::important[这一部分是“交付能力层”]
如果 Part 3 解决的是“主干怎么搭起来”，那 Part 4 解决的就是“这套东西怎么真的跑起来、接起来、验起来、发出去”。
:::

# 读完这一部分，应该多出什么能力

- 能把 AI 从代码生成推进到真实执行
- 能把外部信息、浏览器反馈和终端验证接进控制面
- 能把项目从“看起来完成”推进到“真正可交付”
