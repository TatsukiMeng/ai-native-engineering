# 这一章要解决什么问题

很多 AI 协作在“代码写完”这一刻就戛然而止，好像项目已经结束了。

但真实世界里，联调、验收、回归、变更说明、发布检查、线上反馈，才是决定交付质量的最后一公里。前面如果没有把这些环节也纳入控制面，所谓工作流就还没闭环。

# 这一章会怎么展开

- 先讲联调阶段 CLI、MCP、Playwright、日志和测试该怎么串起来
- 再讲验收和发布时，AI 能参与什么，哪些决策仍然必须由人拍板
- 最后讲一次真实交付结束后，怎样把经验回写成 Rules、Skills、模板和知识资产

:::important[占位说明]
这篇后续会补入联调 checklist、发布前检查模板、回归策略和交付后知识回写流程。
:::

# 读完这一章，读者应该带走什么

- 代码写完只是中场，不是终点
- 真正的工作流必须把联调、验收与发布也串进闭环
- 每一次发布之后，都应该留下能复用的经验资产，而不是只剩一次聊天记录
