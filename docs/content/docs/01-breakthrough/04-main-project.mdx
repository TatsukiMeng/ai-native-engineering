# 为什么这里要先把项目亮出来

这本书不想拿一个过于干净的小 demo 糊弄过去。

因为只要任务足够小、依赖足够少、边界足够简单，很多 AI 工作流看起来都会显得异常顺滑。但那种顺滑，往往经不起真实项目的复杂度。

所以这一章要先把主线项目亮出来：它会涉及前后端、状态边界、自动化执行、外部系统接入、联调交付，甚至后面的治理问题。只有把战场先摆上桌面，后面每一部分才不会显得悬空。

# 这一章会怎么展开

- 先说明这个项目为什么不能用“想到哪写到哪”的方式做
- 再说明它为什么适合作为整本书的主线案例
- 最后把 Part 1 到 Part 5 会如何围着这条主线推进讲清楚

:::note[占位说明]
这篇先按 `OUTLINE.md` 建立章节骨架，后续会继续补入项目背景、边界拆解、关键约束和阶段性验收口径。
:::

# 读完这一章，读者应该带走什么

- 大项目不是拿来证明 AI 很强，而是拿来验证工作流是否可靠
- 主线项目会贯穿全书，不是 Part 3 才突然出现的临时例子
- 后面所有能力都不是“为了炫技”，而是为了把这个项目稳定做完
