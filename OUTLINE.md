# 《AI 原生工程：Vibe Coding 进阶与全域自动化落地》

## 总纲：这本书真正要教会读者什么

- 这不是一本 AI 工具图鉴，也不是一堆热词的科普合集。
- 这本书真正要教会读者的，是怎么把 AI 从一个时灵时不灵的外挂，变成一套能稳定推进大项目的工程工作流。
- 读者读完之后，应该不只会“用一个 AI IDE”，而是能理解：任务怎么拆、Context 怎么喂、Rules 怎么写、Skill 怎么沉淀、工具链怎么接、失败了怎么兜底。
- 再往后一步，不只是个人提效，而是把 AI 用成一个可控、可复用、可治理的生产系统。

## 这次目录重排的关键原则

- **Part 1 不求讲透，只负责起引子。**
  - `Part 1` 的职责不是把每个概念一次讲完，而是先让读者知道：旧脑回路已经不够用了，后面为什么必须升级工作流。
  - 它更像“先给读者装一个最小心智框架”，而不是“概念词典”。

- **项目不再只属于 Part 3，而是成为全书主线。**
  - Part 1 先把项目亮相，让读者知道后面到底要打什么仗。
  - Part 2 提前讲清这个项目会怎么翻车。
  - Part 3 和 Part 4 正式把项目做起来。
  - Part 5 再把这个项目升级成一套可治理的生产系统。

- **不用为了目录完整，硬往项目里塞所有能力。**
  - 有些能力是主线必用，有些能力是按需增强，有些能力是规模上来之后才需要。
  - 这本书应该呈现“能力为什么会在某个阶段出现”，而不是为了秀全家桶强行上强度。

## 全书的项目主线怎么穿

- **Part 1：先把项目抛出来。**
  - 它为什么复杂。
  - 为什么不能再用“想到哪写到哪”的方式做。
  - 为什么必须先建立新的控制面心智。

- **Part 2：先演练它会怎么翻车。**
  - 调试为什么会失控。
  - 测试为什么会自洽。
  - 为什么需要 Rules、Skills、回滚、审计。

- **Part 3：开始做项目主线。**
  - 需求收敛。
  - 架构与数据设计。
  - 后端和前端核心链路落地。

- **Part 4：给这个项目装上更强的手和眼。**
  - MCP。
  - CLI / Harness。
  - Playwright。
  - 外部系统接入与跨端宿主能力。

- **Part 5：把这个项目从“能做完”升级到“能长期跑”。**
  - 成本治理。
  - Swarm 编排。
  - 共享记忆与 Context 治理。
  - 反脆弱设计。

## Part 1 · 破局 —— 这一部分不负责讲透，它负责把读者脑子掰过来

*目标：Part 1 不追求概念讲全、讲深，而是先完成三件事：打破“AI 只是高级补全”的旧认知；给出最小但够用的控制面心智；把后面那个大型项目的主线先埋下去。*

- **1.1 为什么很多人觉得 AI 很笨：其实是工作流还停在补全时代**
  - 从“你写 AI 补”到“你说 AI 做”，中间到底发生了什么。
  - AI 编程为什么不只是模型变强了，而是协作方式、交互方式、开发模式一起变了。
  - Prompt Engineering 为什么不够了，但也远远没死。
  - 这一章只需要让读者先意识到：问题很多时候不在模型，而在工作流。

- **1.2 工具不是信仰：不同产品形态到底在解决什么问题**
  - Native IDE、IDE 插件、CLI、Web Agent、Spec 工具分别擅长什么，不擅长什么。
  - 为什么“找本命工具”越来越不重要，真正重要的是岗位分工与场景匹配。
  - 模型分工、工具分工、成本分工为什么要一起看。
  - 这一章不做产品百科，而是先给读者一个够用的判断框架。

- **1.3 从 Prompt 到 Context 再到 Agent Harness：先把控制面的轮廓看见**
  - Prompt 解决的是任务入口，Context 解决的是材料分发，Harness 解决的是执行闭环。
  - 为什么只会写 Prompt 很快就会撞墙；为什么 Context 多了也不一定更好。
  - `AGENTS.md`、Specs、Skills、运行时信号、验证链路分别处在哪一层。
  - 这一章不要求读者完全吃透，但要先知道后面为什么会反复回到这些概念。

- **1.4 先把主线项目亮出来：为什么这本书要拿一个真正的大项目开刀**
  - 这个项目为什么不能靠“写几个 demo 页面 + 几个接口”糊弄过去。
  - 它会同时碰到用户端、后台端、服务端、数据库、联调、自动化验收、跨端宿主等真实问题。
  - 这也就是为什么后面不会只讲 `SpecDriven`，而是要把整条工作流打通。
  - 这一章的作用是先给读者建立期待：后面不是散点技巧，而是一条持续推进的主线。

## Part 2 · 暗礁 —— 先别急着开工，先知道这东西通常会怎么死

*目标：在项目真正展开之前，先把最容易翻车的地方讲透。这样后面进入实战时，读者看到的不只是“AI 能做到什么”，还能理解“为什么必须这样做，才能不翻车”。*

- **2.1 开发更快了，为什么调试反而更痛苦**
  - AI 生成的代码为什么特别容易变成“黑盒面条代码”。
  - 为什么以前那套断点 + 人肉顺藤摸瓜的方式，在这里经常越调越乱。
  - 可观测性、结构化日志、执行痕迹、分层输出为什么要在生成阶段就前置进去。
  - 这一章不只是讲痛点，而是先给后面的大项目准备排障思路。

- **2.2 不要让 AI 自己给自己打分：最危险的是自洽，不是报错**
  - 为什么不能让 AI 既写业务逻辑，又顺手把测试也写了，然后自己宣布全绿。
  - 空值、权限、边界条件、错误语义、异常分支为什么最容易被糊弄过去。
  - 单测、契约测试、集成测试、E2E、AICR 各自该卡在哪一层。
  - 这一章会让读者明白：后面做项目时，验证链路必须是设计出来的，不是顺手补的。

- **2.3 Rules、Skills 和 SOP：把偶发成功，变成稳定交付**
  - 什么东西应该写进 `AGENTS.md` 或 rules，什么东西更适合沉淀成 Skill。
  - 为什么很多人天天重写 Prompt，本质上是在重复做低价值劳动。
  - 高价值的沉淀，不是“把经验写下来”，而是让下次任务可以直接复用。
  - 这一章会把规则、模板、Skill、流程化约束之间的边界说清楚。

- **2.4 熔断、回滚、审计与小步快跑：AI 时代的工程护栏怎么搭**
  - 为什么一次让 AI 改 1000 行，通常不是勇敢，是找死。
  - Commit-driven generation、小步提交、失败快照、回滚点、决策记录怎么配合。
  - 真正靠谱的护栏，不是禁止 AI，而是保证 AI 失误时损失可控、问题可追。
  - 这一章等于是后面大项目的“施工安全培训”。

## Part 3 · 实战上半 —— 项目正式起盘，先把主干搭起来

*目标：Part 3 开始真正做项目，但不追求一步到位。重点是把最核心的骨架搭起来：需求收敛、系统设计、后端底座、前端主干。先把项目做对，再考虑把它做满。*

- **3.1 项目启动：先收敛需求，不要一上来就让 AI 开写**
  - 怎么把模糊需求收成项目目标、角色边界、核心流程、验收标准。
  - 怎么确定哪些信息该进 Prompt，哪些该进 Spec，哪些该进长期 Context。
  - 怎么把一个大项目拆成 AI 真能消化的阶段、模块和 Ticket。
  - 这一章的重点不是“开始写”，而是“别从第一步就写歪”。

- **3.2 系统设计：架构、数据、接口、模块边界先立住**
  - 从业务流程反推领域模型、数据库设计、接口契约、权限模型和异步任务。
  - 怎样让 AI 输出的不只是“方案文档”，而是能指导后续编码的执行骨架。
  - 哪些地方必须由人拍板，哪些地方可以放给 AI 快速铺开。
  - 这一章会把“人负责 What 和边界，AI 负责 How 和执行”落到真实项目里。

- **3.3 后端主线：让 PostgreSQL、服务层和接口层一起长出来**
  - 从表结构、索引、迁移、校验、鉴权、错误语义到后台管理接口的完整链路。
  - 怎么避免 AI 只会堆 CRUD，而不会处理真实项目里的复杂约束。
  - 哪些能力是主线必做，哪些高级能力应该留到后面按需增强。
  - 这一章不是教 AI 帮忙写接口，而是教读者怎么让后端底座稳得住。

- **3.4 前端主线：不是堆页面，而是先把组件协议和状态边界讲清楚**
  - 从设计稿、组件树、状态流、接口对接到页面增量生成的拆解路径。
  - 为什么整页一把梭通常不靠谱，模块化拆分才更适合 AI 协作。
  - 用户端、中后台、复杂工作台界面分别该怎么设计不同的协作策略。
  - 这一章要解决的是“前端怎么稳定生成”，不是“AI 帮我画个页面”。

## Part 4 · 实战下半 —— 给项目装上手、眼和更完整的执行闭环

*目标：当项目主干已经立住，下一步不是盲目往里加功能，而是补上真正影响交付效率和稳定性的那层能力：CLI、MCP、Playwright、外部系统接入、跨端宿主、联调与发布闭环。*

- **4.1 CLI 与 Harness：终端才是很多高价值任务真正的施工现场**
  - 为什么很多任务最终都要落到脚本、命令、构建、部署和流水线上。
  - 受控执行环境、运行时可见性、机械验证、失败收敛为什么缺一不可。
  - 从“会调用 Bash”到“能在真实环境里持续完成任务”，中间差的到底是什么。
  - 这一章会把项目从“能写代码”推进到“能真正跑流程”。

- **4.2 MCP 与外部系统接入：少让 AI 瞎猜，多让它直接读真实材料**
  - 设计稿、文档、任务系统、数据库、日志平台、知识库分别该怎么接入。
  - 什么应该实时读取，什么应该沉淀成长期 Context，什么应该转成 Spec 或 Skill。
  - 外部信息越多，越要控制噪声、权限和时效性，而不是一股脑全接进来。
  - 这一章解决的是“项目做大之后，信息怎么进工作流”。

- **4.3 Playwright 与多模态执行：让 AI 真正有眼睛和手**
  - 浏览器自动化如何承担 UI 验证、流程回归、截图对比和定位问题。
  - 为什么 Playwright 的价值不只是测试，而是可观察、可复查、可回放。
  - 当前端问题不再只靠“你描述一下页面错哪了”，协作方式会怎么变化。
  - 这一章会把前端联调和验收真正串进项目闭环里。

- **4.4 跨端与宿主：当项目不只跑在浏览器里，控制面要怎么升级**
  - 微前端、多应用接入、宿主通信、桌面端封装这些边界怎么先定义再落代码。
  - Tauri、Wujie 这类场景里，AI 最容易犯什么错，应该怎么约束它。
  - 当项目开始碰到本地能力、文件系统、IPC、打包和平台差异时，控制面该怎么升级。
  - 这一章不是为了炫跨端，而是把真实项目的边界补全。

- **4.5 联调、验收与发布：真正的工作流，不到上线那一刻都不算完**
  - CLI、MCP、Playwright、日志、测试、AICR 在联调阶段怎么串起来。
  - AI 怎么参与 bugfix、回归、变更说明、发布检查与交付交接。
  - 出现偏差之后，怎么把本次经验回写成 Rules、Skill、模板和知识资产。
  - 这一章会让项目主线真正闭环，而不是停在“代码写完了”。

## Part 5 · 终局 —— 当项目做完以后，真正的问题才刚刚开始

*目标：把话题从“这个项目怎么做完”抬升到“这套工作流怎么长期跑”。这里讨论的不是技巧堆叠，而是成本治理、多 Agent 编排、共享记忆、反脆弱设计和开发者角色迁移。*

- **5.1 成本治理：真正贵的不是 token，而是低质量上下文和无效返工**
  - 为什么很多人觉得 AI 贵，本质上不是模型太贵，而是工作流太浪费。
  - 模型分层、任务分层、上下文预算、额度窗口、重试策略怎么一起设计。
  - 成本治理为什么必须前置，而不是等账单爆了才开始补锅。
  - 这一章会把“省钱”这件事从抠门，变成工程调度问题。

- **5.2 Swarm 编排：多 Agent 协作不是多开几个窗口那么简单**
  - PM Agent、架构 Agent、开发 Agent、测试 Agent、发布 Agent 分别该做什么。
  - 多 Agent 系统里，任务拆分、职责边界、共享上下文、冲突控制怎么设计。
  - 为什么没有编排层，多 Agent 只会把混乱放大得更快。
  - 这一章会把“虚拟员工团队”从比喻拉回真实工程系统。

- **5.3 共享记忆与 Context 治理：系统该记住什么，又该忘掉什么**
  - 为什么多轮任务之后，AI 很容易被旧上下文、旧实现、旧结论带偏。
  - 共享记忆区、决策日志、知识库、索引系统分别适合存什么。
  - 顺序、频率、重复引用、时效性、迁移期共存这些因素会怎么影响 AI 判断。
  - 这一章会把“Context as Code”真正落到治理问题上。

- **5.4 反脆弱设计：让系统在失败里收敛，而不是在幻觉里互相放大**
  - 开发 Agent 和测试 Agent 互相甩锅时，熔断点该设在哪里。
  - API 幻觉、错误引用旧路径、上下文污染为什么会在多 Agent 系统里级联放大。
  - 交叉验证、失败归因、决策复盘、记忆更新怎么形成真正的学习闭环。
  - 这一章要解决的，不是“怎么不出错”，而是“出错后系统怎么变得更稳”。

- **5.5 架构师的升级：以后最值钱的，不只是会写代码的人**
  - AI 时代，开发者的核心价值会从“写实现”逐渐转向“设计规则、流程和容错底线”。
  - Code Review 会越来越像决策审查、风险审查和验证链路审查。
  - 人类不会从系统里消失，但会从执行者更明显地转向系统设计者和最终责任人。
  - 这一章会给整本书一个真正的收束：我们到底该成长成什么样的人。

## 最后一句话

- 这本书最终要回答的，不是“哪个工具最强”，而是：**怎样把 AI 从一个看起来很猛、用起来很玄的东西，变成一套你真敢拿去做大项目的工程系统。**
